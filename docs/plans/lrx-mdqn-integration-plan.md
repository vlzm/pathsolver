## 1. Текущее состояние (что уже есть и чего нет)

### Репозиторий — уже общий решатель Кэли-графов

Ключевой факт: в `pathsolver/` **нет кода, специфичного для кубика**. Пазл описывается тройкой
данных, которая целиком проходит через `train.py`/`test.py`:

- `generators/pXXX.json` — `{"actions": [[перестановка], ...], "names": [...]}`
- `targets/pXXX-tYYY.pt` — 1-D тензор «собранного» состояния `V0`
- `datasets/pXXX-tYYY-rnd.pt` — 2-D тензор скрамблов для тестирования

**LRX уже зарегистрирован** как группы `034`–`037` (n = 10, 15, 20, 25):

| id | что есть | чего нет |
|---|---|---|
| p034 (n=10) | generators, target, dataset, **веса (128 эпох), ONNX** | UI |
| p035 (n=15) | generators, target, dataset, **веса (64 эпохи), ONNX** | UI |
| p036–p037 (n=20, 25) | generators, targets, datasets, строки в README и `traintest-tab4-rnd.sh` | весов, ONNX, UI |

Помимо README-таблицы в репозитории оказались ещё две LRX-группы: **p038 (n=30)** и
**p039 (n=35)** — с генераторами, targets и датасетами, но не описанные в README.
p038 фигурирует в `traintest-tab4-rnd.sh` (K_max 300). Для воспроизведения S₂₈ из
статьи это ближайшие кандидаты (см. открытый вопрос 3).

Основные модули:

- `pathsolver/model.py` — `PathSolver(nn.Module)`: one-hot → MLP (+residual) → скаляр
  (регрессия diffusion distance). `num_classes = torch.unique(V0).numel()` — для куба 6,
  для LRX = n. Модель уже подходит для LRX без изменений.
- `pathsolver/trainer.py` — `Trainer`: non-backtracking random walks
  (`do_random_step` банит обратный ход через `inverse_moves`),
  `generate_random_walks(k, K_min, K_max)` → (X, Y=число шагов), supervised-цикл `run()`
  с регенерацией датасета каждую эпоху.
- `pathsolver/searcher.py` — `Searcher`: векторизованный beam search с хеш-дедупликацией
  (`hash_vec`), **восстановлением пути** (`tree_idx`/`tree_move`) и детекцией стагнации.
  Функционально покрывает оба варианта beam search из ноутбука.
- Конфиг — чистый argparse; гиперпараметры запуска сохраняются в
  `logs/model_<name>_<id>.json` и перечитываются `test.py`.

### Ноутбук — что из него нужно портировать

`notebooks/lrx-cayleypy-rl-mdqn.ipynb` (Kaggle, standalone, ~87 ячеек):

1. **Генераторы LRX**: `get_LRX_moves(n)` → L (цикл. сдвиг влево), R (вправо),
   X (транспозиция позиций 0,1).
2. Random walks: simple и non-backtracking (бан по истории хешей глубины k).
3. MLP + supervised warm-up (MSE на числе шагов RW) — *аналог уже есть в `Trainer`*.
4. **MDQN-дообучение** (ячейка 68) — главное, чего нет в репозитории:
   - сгенерировать RW-датасет (X, y_rw);
   - для всех X посчитать соседей (`get_neighbors`) и Bellman-цель
     `y_bellman = 1 + min_соседи model(neighbor)`;
   - клиппинг: `y = min(y_bellman, y_rw)` (истинное расстояние ≤ числа шагов RW),
     `y = clamp_min(y, 1)`, граничное условие `y[стартовые состояния] = 0`;
   - опционально округление (`flag_dqn_round`, в статье не даёт улучшения);
   - градиентный шаг MSE(model(X), y).
   - Результат статьи: warm-up 30 эпох + MDQN 200 эпох — лучшая комбинация для S₂₈
     (медиана длины пути 402 vs 650 без DQN vs None у чистого DQN).
5. Beam search torch (без пути) и «версия Фиронова» (с путём) + проверка корректности
   пути — *покрыто `Searcher`*, но есть LRX-специфичные эвристики, которых нет:
   - `ban_p0_p1_transposition_if_p0_lt_p1_ie_already_sorted` — не применять X,
     если p0 < p1 (эвристика Фиронова, опциональная, по умолчанию False);
   - запрет L сразу после R и наоборот (non-backtracking по имени хода) —
     в `Searcher` non-backtracking реализован иначе (хеш-дедупликация состояний),
     этого достаточно.
6. **«Самый длинный элемент» LRX** (ячейка 72): перестановка-гипотеза
   `(0,1)(n-1,2)(n-2,3)...` с ожидаемой длиной разложения n(n−1)/2 (OEIS-A186783) —
   полезно добавить как утилиту/спец-скрамбл для бенчмарка.

### Расхождения ноутбук ↔ репозиторий (решения)

| Вопрос | Ноутбук | Репозиторий | Решение |
|---|---|---|---|
| Набор генераторов | {L, R, X} (3 шт.) | p034: {L, L', X, X'} (4 шт., причём X' ≡ X — дубликат) | Оставить JSON как есть (R = L', совместимо с `generate_inverse_moves`). Дубликат X' слегка расширяет ветвление beam — можно почистить позже, отдельным решением. |
| Non-backtracking RW | бан по истории хешей глубины k | бан обратного хода (глубина 1) | Оставить механизм репозитория для warm-up; для MDQN он тоже достаточен (статья требует лишь non-backtracking). Опционально: параметр глубины истории как улучшение. |
| Модель | простой MLP [128] | PathSolver (hd1/hd2/residual) | Использовать PathSolver; размеры сети задавать через существующие CLI-параметры. |
| Beam search | 2 варианта, эвристика Фиронова | Searcher (с путём) | Использовать Searcher; эвристику Фиронова добавить опциональным флагом (выкл. по умолчанию). |

## 2. Архитектурные решения

1. **Не создавать класс «окружения»** — data-driven-схема репозитория уже абстрагирует
   пазл. LRX не требует нового кода загрузки.
2. **MDQN — расширение тренера**, а не отдельная копия пайплайна: новый модуль
   `pathsolver/dqn.py` (или методы в `Trainer`), переиспользующий `do_random_step`,
   `generate_random_walks`, оптимизатор и схему сохранения весов.
3. **Модель остаётся скалярным регрессором** (`output_dim=1`, `forward().flatten()`) —
   это сохраняет совместимость с `Searcher.pred_d`, ONNX-экспортом и JS `batchPredict`.
4. **ONNX-экспорт оформить кодом** (сейчас это недокументированный ручной шаг):
   `export_onnx.py` c input `input` (int64, [B, state_size], dynamic batch) и output
   `output` — ровно контракт `ui/web/cube/worker.js`.
5. **UI: обобщить worker, а не форкать вслепую**: `worker.js` куба почти полностью
   generic (gather-семантика ходов, beam, ONNX) — вынести в конфиг `STATE_SIZE`,
   `solvedState`, пути к `model.onnx`/`moves.json`; LRX-страница получает свою
   визуализацию (ряд пронумерованных плиток), но переиспользует solver-worker.

## 3. Этапы работ

### Этап 1 — MDQN в пакете `pathsolver` ✅ ГОТОВ

**Файлы:** `pathsolver/dqn.py` (новый), `pathsolver/trainer.py`, `pathsolver/__init__.py`, `train.py`.

**Как реализовано:**

- `get_neighbors(states, all_moves, batch_size)` вынесен в `pathsolver/utils.py`;
  `Searcher.get_neighbors` делегирует ему (дублирования нет).
- `pathsolver/dqn.py::bellman_targets(net, X, y_rw, all_moves, V0, batch_size, flag_round)` —
  соседи разворачиваются **по чанкам** (`batch_size // n_gens`), а не материализуются
  целиком: тензор `[N, n_gens, state_size]` на 1M состояний — это лишние сотни МБ.
- `Trainer.run_dqn(num_epochs, walkers_num, flag_round)` переиспользует
  `generate_random_walks`, **тот же оптимизатор** (моменты Adam переносятся из warm-up),
  счётчик эпох и схему сохранения. Лог — `logs/dqn_<name>_<id>.csv` отдельным файлом,
  чтобы не ломать схему существующего `train_*.csv`.
- Граничное условие реализовано как маска `(X == V0).all(dim=1)`, а не срез
  `y[:k] = 0` как в ноутбуке: `generate_random_walks` шаффлит батч, поэтому позиционный
  срез был бы неверен.
- CLI: `--epochs_dqn` (0 по умолчанию), `--dqn_walkers` (0 → 1/10 от warm-up),
  `--dqn_round`. Попадают в `logs/model_*.json` автоматически.

**⚠️ Отклонение от плана — именование весов.** План предлагал
`weights/<name>_<id>_dqn_e<epoch>.pth`, но `test.py:100` жёстко собирает путь как
`{name}_{id}_e{epoch:05d}.pth` — с суффиксом `_dqn` веса стали бы незагружаемыми.
Поэтому DQN **продолжает сквозную нумерацию эпох** в том же формате. Неоднозначности
нет: warm-up-only и warm-up+MDQN — разные прогоны с разными `id`.

1. Реализовать `get_neighbors(states, all_moves)` (батчевое применение всех ходов —
   в `Searcher` уже есть аналог, вынести в `pathsolver/utils.py`, чтобы не дублировать).
2. Реализовать MDQN-цикл (референс — ячейка 68 ноутбука):
   ```
   для каждой dqn-эпохи:
       X, y_rw = random walks (как в Trainer, k = walkers_num_dqn)
       neigb   = get_neighbors(X, all_moves)
       y_bell  = 1 + min_by_move model(neigb)        # батчами, no_grad
       y       = clamp_min(min(y_bell, y_rw), 1)
       y[стартовые (solved) состояния] = 0
       if flag_round: y = round(y)
       shuffle; MSE-шаги оптимизатора
   ```
3. CLI в `train.py`: `--epochs_dqn` (default 0 → поведение не меняется),
   `--dqn_walkers` (или доля от основных), `--dqn_round` (flag).
   Все новые параметры писать в `logs/model_<name>_<id>.json`.
4. Схема запуска: warm-up (существующий `Trainer.run`) → MDQN → сохранение весов
   `weights/<name>_<id>_dqn_e<epoch>.pth` (+ по степеням двойки, как сейчас).
5. Санити-тест: для n=10 (p034) сравнить Spearman-корреляцию предсказаний с истинными
   расстояниями (BFS до ~10⁶ состояний, как в ноутбуке) до/после MDQN — ожидаем
   улучшение (в статье для n=10: 0.923 → 0.938–0.939).

**Критерий готовности:** ✅ выполнен. `python train.py --group_id 34 --target_id 0 --K_max 50 --epochs N --epochs_dqn M` обучает end-to-end.

Идентичность при `--epochs_dqn 0` проверена не на глаз: собран git-worktree на HEAD,
прогнано сид-фиксированное обучение в обеих версиях, сравнён SHA-256 state_dict —
совпадает побайтово (`c66447bb…`).

Санити-тест по Spearman **проведён для warm-up** (BFS по всем 10! = 3 628 800 состояниям,
см. раздел 6), но **сравнение до/после MDQN не выполнено** — это остаётся за этапом 3.

## 6. Эмпирические факты (получены при реализации, 2026-07-21)

### Полный BFS для LRX n=10

Весь граф Кэли обойдён за 8 с на CPU (все 10! = 3 628 800 состояний). Состояния
индексируются **рангом кода Лемера** — `c[i] = #{j > i : p[j] < p[i]}`,
`rank = Σ c[i]·(N−1−i)!` — что даёт биекцию в `[0, N!)` и позволяет держать расстояния
в плоском массиве вместо хеш-множества.

- **Диаметр ровно 45 = n(n−1)/2** — совпадает с гипотезой OEIS-A186783 из ячейки 72 ноутбука.
- На глубине 45 **ровно одно** состояние — тот самый «длинный элемент».

> Грабли: первая версия ранжирования сворачивалась по неверной оси
> (`sum(dim=1)` вместо `sum(dim=2)`), давая не код Лемера, а число инверсий слева.
> Коллизии обрывали BFS на 13 062 состояниях — то есть **тихо выглядели как сошедшийся
> обход**, а не как ошибка. Полезно проверять BFS на `seen == n!`.

### Качество warm-up-модели (p034, n=10)

Корреляция предсказаний с истинным расстоянием на 20k случайных состояний:

| чекпоинт | Spearman | Pearson |
|---|---|---|
| random-init | −0.020 | −0.020 |
| эпоха 4 | +0.890 | +0.888 |
| эпоха 16 | +0.936 | +0.910 |
| эпоха 64 | **+0.951** | +0.916 |

В статье для n=10: warm-up 0.923, MDQN 0.938–0.939. Наш warm-up уже выше — вероятно,
за счёт более крупной сети PathSolver (1024/256 + residual) против MLP [128] в ноутбуке.
**Это поднимает планку для этапа 3: на n=10 MDQN может не дать видимого выигрыша,
и осмысленнее проверять его на n≥20.**

### Три ловушки при оценке качества

1. **Лосс — не сигнал.** MSE выходит на плато ~72.6 уже к 3-й эпохе и дальше не движется,
   хотя Spearman растёт с 0.890 до 0.951. Причина — неустранимый шум меток: длина
   случайного блуждания систематически больше истинного расстояния. По лоссу нельзя
   ни выбирать чекпоинт, ни судить о сходимости.
2. **Широкий beam маскирует качество эвристики.** При B=4096 на n=10 чекпоинты эпох
   4, 16, 32 и 64 дают **побайтово одинаковые** решения: граф мал, поиск вырождается
   почти в BFS. Выглядит так, будто модель не подключена.
3. **Нужен контроль необученной моделью.** Он и снимает подозрение из п.2:
   random-init решает **0/5 даже при B=256**, тогда как обученная — 5/5 уже при B=8.
   Эвристика критична; решения просто упираются в оптимум, ниже которого не прыгнешь.

### Прочее

- **MPS (Apple Silicon) не даёт выигрыша:** 17.7 с против 21.0 с на эпоху. Генерация
  random walks на MPS *медленнее* CPU и съедает почти весь выигрыш от обучения.
  16% не стоят нового кодового пути через непроверенный бэкенд; `train.py` по-прежнему
  знает только про cuda и cpu.
- Скрипты валидации (BFS, Spearman, сравнение beam-ширин, решение через ONNX) писались
  во временный каталог и в репозиторий не попали. Если они нужны как воспроизводимые
  бенчмарки — их стоит оформить в `scripts/` (кандидат в этап 2 или 6).

### 8.1. Полный BFS для n=10 переиспользован как точная модель

BFS из раздела 6 воспроизведён (все 3 628 800 состояний, диаметр 45, на нём ровно

которые слабая эвристика замаскировала бы.

- длинный элемент решён ровно за **45** ходов (= диаметр), путь перепроигран по сырым

Отдельно тот же `vm`-приём применён к **`ui/web/lrx/script.js`** целиком (со stub-ом DOM
и Worker, замкнутым на настоящий `solver-core.js`): проверено, что пресет «длинный
элемент» равен `[1,0,9,8,7,6,5,4,3,2]`, что `applyMove` страницы совпадает с gather-
семантикой воркера, что L∘R и X² — тождества, и что после анимации решения страница
реально приходит в `[0..9]`.
